# dex2dot0/skills

Open-source agent skills for the [skills.sh](https://skills.sh) ecosystem.

## Disclaimers

These skills use a heuristic and local agent scoped execution to estimate token costs savings, not real token counts. Substitute a model-specific tokenizer if you need precision and be wary of assuming token savings until proven otherwise. This is a new repository with new skills that have not been battle tested. Use at your own risk. No warranties expressed or implied.

## Skills

### low-token-copy

Directs agents to perform file copies, moves, and renames via shell ops (`mv`, `cp`, `git mv`) and to update references with `Grep` + `StrReplace` instead of read-then-write patterns — saving tokens on bulk reorgs, large-file moves, and import fixups.

Install:

```bash
npx skills add dex2dot0/skills@low-token-copy
```

See [`skills/low-token-copy/SKILL.md`](skills/low-token-copy/SKILL.md).

### low-token-edit

Directs agents to perform in-file edits via targeted patches (`StrReplace` or equivalent) instead of full-file `Write` — including for markdown, docs, configs, and other structured text — and explains why patches are the safer primitive, not the riskier one.

Install:

```bash
npx skills add dex2dot0/skills@low-token-edit
```

See [`skills/low-token-edit/SKILL.md`](skills/low-token-edit/SKILL.md).

## Install isolation

The Skills CLI (`npx skills add owner/repo@skill-name`) fetches **only the matched `skills/<skill-name>/` subtree**. Everything else in this repo — `tests/`, `tests/fixtures/`, `.github/`, root-level docs — is repo infrastructure and is **never installed** to a user's machine. This is reinforced by `.gitattributes` `export-ignore` entries so the same exclusions apply to `git archive` tarballs.

## Cost models

Each skill ships with a deterministic **token-cost model** (not an empirical benchmark — see methodology below). Zero dependencies, Node 18+.

### Methodology and caveats

These scripts are **estimates from a fabricated cost model**, not measurements of real agent runs. Read them as order-of-magnitude indicators, not precise numbers.

**What the model does:**

- Estimates token counts using `~4 chars/token` (accurate to ±20–25% vs real tokenizers like cl100k / o200k).
- Adds a fixed `20-token` envelope per tool call to approximate JSON / schema overhead.
- Reports two B variants: **B opt** (perfect skill execution) and **B real** (B opt + ~30% slip overhead from verification reads, pre-reads, and StrReplace retries — the empirically observed agent imperfection rate from this repo's own session).
- Includes **edge / unfavorable scenarios** (tiny files, ambiguous-anchor edits, scattered changes) where B's advantage narrows or inverts.

**What the model does _not_ measure:**

- Real tokenizer counts. Substitute a model-specific tokenizer if you need precision.
- Wall-clock latency or API round-trip cost.
- Pricing asymmetry. Output tokens typically cost **3–5× input tokens** at major providers, so output savings (visible in the per-call deltas) are worth more than the headline totals suggest.
- Prompt-cache effects. With cached prompts (e.g. Claude prompt caching), the cost of additional `Read` calls collapses, which would narrow Approach A's deficit.
- Real-world agent variance. Different agents, harnesses, and IDE tools execute these patterns differently. Approach A in the model is a worst-case reconstruction of what some agents do by default; not all agents do this.
- Confirmation bias. The skill, the model, and this analysis were authored by the same agent. Treat the conclusions as a hypothesis to be falsified, not a proof.

**Robustness:**

The qualitative conclusions hold across plausible tweaks (envelope ∈ \[5, 100\], chars/token ∈ \[3, 5\]) for the clear-win scenarios. They **do not hold** for the edge scenarios — see those rows for cases where B's advantage is small or zero.

### Cost model: low-token-copy

Token-cost comparison of two approaches to moving a directory and updating its references:

- **Approach A** — `Read` + `Write` + `Delete` for each moved file, plus `Read` + `StrReplace` for each consumer (a potentially common agent default).
- **Approach B** — single `git mv directory` + `Grep` to find consumers + `StrReplace` with `replace_all` (the pattern this skill teaches).

Reproduce:

```bash
node tests/compare-approaches.mjs            # print results
node tests/compare-approaches.mjs --write    # also update this table
```

The harness scans the real fixture at `tests/fixtures/sample-module/` for the first row, and uses synthetic file sizes for the rest.

- **Approach A** — `Read` + `Write` the whole file (the common agent default for markdown and docs).
- **Approach B** — `StrReplace` patches scoped to the actual change (the pattern this skill teaches).

node tests/compare-edits.mjs            # print results
